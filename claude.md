# 프로젝트: 2026년 인사평가프로그램

## 작업규칙(절대명제)
1. 우선 작업할 내용을 md 파일로 작성
    - 폴더 위치는 works폴더의 하위 tasks폴더
    - ./works/tasks
    - 작업지시서 작성시 작업의 규모에 따라 간소화해서 작업지시서를 작성
      - TEMPLATE_README.md 를 참조해서 템플릿을 활용
      - 간단한 작업(1-2개 파일, 50줄 미만): TEMPLATE_simple.md
      - 복잡한 작업(3개 이상 파일, 기능 추가): TEMPLATE_detailed.md
      - 판단이 어려운 경우: TEMPLATE_README.md의 6가지 질문 참고      

    ### 📁 템플릿 파일 위치

    ```
    works/
    ├── tasks/
    │   ├── TEMPLATE_simple.md  
    │   ├── TEMPLATE_detailed.md
    │   └── TEMPLATE_README.md     
    ```    

2. 작업할 내용이 많으면 파일을 분할해서 작성
3. 생성된 작업지시서를 개발자와 같이 검토
4. 검토가 완료된 작업지시서를 기반으로 작업
  - 작업지시서의 명명규칙은 날짜_순번_간략한 작업명.md 로 작성
  - 예시 : 20251014_01_css_modify.md
5. 서버의 실행등 테스트는 개발자가 진행함
   - 테스트할 항목만 알려주면 됨
6. 추가적인 권장사항이나 개선점이 있을 경우
   - 먼저 개발자에게 확인 후 승인을 받고 진행함
   - 진행시 작업내용에 대한 작업지시서를 작성
   - 작성된 작업지시서를 기반으로 작업

- 주의사항 : 파일 생성 및 수정은 개발자의 요청에 의해서만 시행
- 주의사항 : 임의로 판단해서 수정은 절대안됨

## 이슈관리(절대명제)

### 1. 이슈 파일 구조
```
works/
├── issue.md                          # 이슈 인덱스 (전체 이슈 목록 및 현황)
├── issues/                           # 개별 이슈 파일 폴더
│   ├── 001_swagger_ui_error.md
│   ├── 002_api_timeout.md
│   ├── 011_database_design.md
│   └── ...
└── tasks/
    └── [작업지시서들...]

```

### 2. 새 이슈 생성 규칙
1. **이슈 번호 확인**: issue.md에서 마지막 이슈 번호 확인
2. **개별 파일 생성**: `works/issues/{번호}_{영문제목}.md` 형식으로 생성
   - 예시: `works/issues/012_phase1_implementation.md`
   - 번호는 3자리 (001, 002, ..., 012)
   - 영문 제목은 소문자와 언더스코어 사용
3. **인덱스 업데이트**: issue.md의 이슈 목록에 새 이슈 추가
4. **템플릿 사용**: 아래 템플릿으로 개별 파일 작성

### 3. 이슈 인덱스 (issue.md) 관리
- **역할**: 전체 이슈의 목록, 상태, 관계를 한눈에 파악
- **내용**:
  - 이슈 현황 통계 (총 개수, 완료/진행/보류)
  - 진행 중인 이슈 목록
  - 최근 완료된 이슈 목록 (최근 5개)
  - 전체 이슈 목록 (접을 수 있는 형태)
  - 이슈 간 관계도 (선택사항)
- **업데이트 시점**:
  - 새 이슈 생성 시
  - 이슈 상태 변경 시 (진행 중 → 완료 등)

### 4. 개별 이슈 파일 작성법
- **파일명**: `{번호}_{영문제목}.md`
- **내용**: 아래 템플릿 사용
- **관련 이슈 링크**: 상대 경로로 다른 이슈 파일 링크
- **작업지시서 링크**: 상대 경로로 작업지시서 링크

### 5. 이슈 파일 템플릿
```markdown
# 이슈 #{번호}: {제목}

**날짜**: YYYY-MM-DD
**상태**: 진행중 / 완료 / 보류
**우선순위**: 높음 / 중간 / 낮음
**관련 이슈**: [#XXX](XXX_title.md), [#YYY](YYY_title.md)
```

### 6. 연속 작업 시 절차
1. **issue.md 확인** → 최근 이슈 및 진행 상황 파악
2. **개별 이슈 파일 읽기** → 상세 내용 및 진행 사항 확인
3. **관련 이슈 링크 따라가기** → 선행 작업 컨텍스트 이해
4. **작업지시서 확인** → 구체적인 작업 내용 파악
5. **작업 진행** → 진행 사항 체크리스트 업데이트
6. **완료 후** → 이슈 파일과 issue.md 상태 업데이트

### 7. 이슈 상태 관리
- **진행중**: 현재 작업 중인 이슈
- **완료**: 작업이 완전히 끝나고 개발자 피드백까지 받은 이슈
- **보류**: 일시적으로 중단되었거나 추후 처리할 이슈

### 8. 주의사항
- 작성된 이슈사항을 인지하고 추후 작업시 반영해서 작업
- 관련 이슈는 반드시 링크로 연결하여 추적 가능하게 관리
- 개별 파일이 너무 길어지지 않도록 핵심 내용만 기록
- 상세한 기술 문서는 별도 작업지시서로 분리

## 사용자 정의 도구 (절대명제)

**중요**: 본 프로젝트는 `.claude` 폴더에 사용자 정의 Agent와 Skill이 구성되어 있습니다.

### 세션 시작 시 필수 절차

1. **세션 시작 시 자동으로 `.claude/README.md` 읽기**
   - 위치: `.claude/README.md`
   - 이 파일에 모든 Skills와 Agents의 최신 목록 및 사용법이 정의되어 있음
   - **절대 생략하지 말 것** - 도구를 모르면 비효율적인 수동 작업을 하게 됨

2. **도구 우선 사용 원칙**
   - 작업지시서 작성 필요 → Skill 확인 후 사용
   - DB 관련 작업 → Skill 또는 Agent 확인 후 사용
   - 이슈 관리 필요 → Skill 확인 후 사용
   - 분석 또는 검증 필요 → Agent 확인 후 사용

3. **업데이트 주기**
   - Skills와 Agents는 지속적으로 추가/변경됨
   - 새 세션 시작 시 `.claude/README.md` 재확인 필수

### 주의사항

- **단일 소스 원칙**: 도구 정보는 `.claude/README.md`에서만 관리
- **절대 수동 작업 금지**: 사용 가능한 Skill/Agent가 있으면 반드시 활용
- **상세 문서**: 각 도구의 `agent.md` 또는 `skill.md` 파일 참조

### 4. Custom Agents 사용법

#### Agent 1: checklist-generator (체크리스트 생성)

**용도**: 동기화할 파일 목록 및 복사 절차 가이드 생성

**사용 시점**: 현재 프로젝트 작업 완료 후

**명령어**:
```
"checklist-generator로 동기화 체크리스트 만들어줘"
```

**출력**:
- 파일: `works/sync-checklists/YYYYMMDD_HHMM_sync_checklist.md`
- 내용: 변경 파일 목록, 복사 절차, 체크박스

**예시**:
```markdown
# 동기화 체크리스트

**생성일시**: 2026-01-22 15:30
**변경 파일**: 5개

## 복사 절차

### 1. Login.razor
- [ ] VSCode에서 복사
- [ ] Visual Studio에 붙여넣기
- [ ] 빌드 확인

(반복)
```

---

#### Agent 2: sync-validator (동기화 검증)

**용도**: 복사 후 두 프로젝트 간 차이 검증

**사용 시점**: 수동 복사 완료 후

**명령어**:
```
"sync-validator로 동기화 확인해줘"
```

**출력**:
- 파일: `works/sync-reports/YYYYMMDD_HHMM_validation_report.md`
- 내용: 통과/차이/누락 파일 리포트

**예시**:
```markdown
# 동기화 검증 리포트

✅ 통과: 4개
⚠️ 차이: 1개
  - UserRepository.cs (Line 256 불일치)
❌ 누락: 0개

권장 조치: UserRepository.cs 재복사
```

### 6. 주의사항

1. **자동 동기화 금지**: Agent는 파일을 수정하지 않음 (읽기 전용)
2. **수동 복사 원칙**: 개발자가 직접 확인하며 복사
3. **빌드 필수**: 복사 후 Visual Studio에서 반드시 빌드 확인
4. **Git 분리**: 두 프로젝트는 독립된 Git 저장소 사용
5. **테스트 파일 제외**: `tests/` 폴더는 동기화하지 않음
